HIGH

1. Protected-path rejection is bypassable for non-existing protected children when the output path uses a symlink.

Evidence:
- agentic-system-builder/scripts/scaffold-agent-system.mjs:257 normalizes outputDir with resolve(outputDir), not a canonical realpath for non-existing paths.
- agentic-system-builder/scripts/scaffold-agent-system.mjs:270 checks whether the normalized path is inside agentic-system-builder or .omo.
- agentic-system-builder/scripts/scaffold-agent-system.mjs:297 only calls existingRealPath(outputDir) after the first path-safety assertion and only when the path already exists.
- Fresh temp probe created a symlink to the temp workspace, then ran:
  node agentic-system-builder/scripts/scaffold-agent-system.mjs --domain library --out <work-link>/agentic-system-builder/subdir --force
  It exited 0 and wrote README.md into the protected source subtree through the symlink.
- The canonical spelling of the same nested protected paths rejects correctly. The bypass depends on path spelling, not user intent.

Impact:
- The current code still allows scaffolding into protected workspace internals when the target child does not already exist and the output path reaches the workspace through a symlink.
- Existing protected targets are safer because realpath(outputDir) is checked when they exist, but new protected children are not covered.
- This violates the requested "dangerous paths remain rejected" criterion and the script's own line 272 intent to refuse output inside protected workspace.

Required fix:
- Canonicalize the nearest existing ancestor before deciding whether a non-existing output path is inside ROOT, agentic-system-builder, .omo, or generated.
- Apply the same canonical path basis to ROOT/protected dirs and requested output paths before any mkdir/write/rm.
- Add a regression probe for symlinked workspace/output path spelling, including a non-existing protected child under agentic-system-builder/ and .omo/.
